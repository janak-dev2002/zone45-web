# Database Schema — ZoneForty5 Website

> STATUS: DRAFT (architecture skeleton). Database Agent owns the final schema.
> Author of this draft: Architecture Agent (Claude Opus 4.7), 2026-05-25
> Database Agent may add columns/indexes for performance, but MUST NOT change names or types of fields used by `api-contracts.md` without updating both files and `change-log.md`.

This is the source of truth for the data layer. Database Agent translates this into `node-pg-migrate` migrations in `backend/src/db/migrations/`. Backend Agent writes the query layer against these tables and types.

---

## 1. Conventions

- **DBMS:** PostgreSQL 16.
- **Migration tool:** `node-pg-migrate` — chosen so migrations are SQL-shaped, reversible, and ORM-independent.
- **Naming:** `snake_case` for tables and columns; plural table names; PK column named `id`; FK columns named `<referenced_singular>_id`.
- **Primary keys:** `uuid` generated by `gen_random_uuid()` (requires the `pgcrypto` extension; enable in the first migration). UUIDs let admin UIs include the id in URLs without leaking row counts.
- **Timestamps:** every content table has `created_at` and `updated_at` (both `timestamptz NOT NULL DEFAULT now()`). `updated_at` is maintained by an `AFTER UPDATE` trigger (one shared trigger function `set_updated_at()` registered per table).
- **Booleans:** `boolean NOT NULL DEFAULT false`.
- **Soft delete:** NOT used at MVP. Deletes are hard. Documented as a future revisit.
- **Character set:** UTF-8 (Postgres default since cluster init with `--encoding=UTF8`).
- **Collation:** `en_US.UTF-8` (default). Slug uniqueness uses citext columns so case-only differences collide.


## 3. Shared helpers

The first migration also creates the shared `updated_at` trigger function:

```sql
CREATE OR REPLACE FUNCTION set_updated_at() RETURNS trigger
LANGUAGE plpgsql AS $$
BEGIN
  NEW.updated_at := now();
  RETURN NEW;
END;
$$;
```

Every table with `updated_at` gets a trigger:
```sql
CREATE TRIGGER trg_<table>_updated_at
BEFORE UPDATE ON <table>
FOR EACH ROW EXECUTE FUNCTION set_updated_at();
```

## 4. Entity definitions

### 4.1 `admin_users`

The admin panel's user table. Modelled as a table (not a single env-driven row) so a second admin can be added later without a migration. At MVP there is exactly one row, bootstrapped from env vars on first boot.

```sql
CREATE TABLE admin_users (
  id              uuid        PRIMARY KEY DEFAULT gen_random_uuid(),
  email           citext      NOT NULL UNIQUE,
  password_hash   text        NOT NULL,         -- Argon2id encoded string
  name            text        NOT NULL,
  last_login_at   timestamptz NULL,
  created_at      timestamptz NOT NULL DEFAULT now(),
  updated_at      timestamptz NOT NULL DEFAULT now()
);
```

- `email` is `citext` so case differences collide (no two admins with `Admin@x` and `admin@x`).
- `password_hash` holds the full Argon2id encoded string (`$argon2id$v=19$m=65536,t=3,p=4$...`); parameters are embedded so verification works after parameter changes.
- `name` is what the admin UI displays in the top bar; defaults to `"Admin"` in the seed if `ADMIN_NAME` env var is unset.
- No FK to other tables at MVP (see architecture §5 decision note).

### 4.2 `portfolio_projects`

```sql
CREATE TABLE portfolio_projects (
  id                uuid        PRIMARY KEY DEFAULT gen_random_uuid(),
  slug              citext      NOT NULL UNIQUE,
  title             text        NOT NULL,
  description       text        NOT NULL,
  body              text        NOT NULL,
  tech_stack        jsonb       NOT NULL DEFAULT '[]'::jsonb,
  outcome           text        NULL,
  project_url       text        NULL,
  cover_image_url   text        NULL,
  sort_order        integer     NOT NULL DEFAULT 0,
  published         boolean     NOT NULL DEFAULT false,
  created_at        timestamptz NOT NULL DEFAULT now(),
  updated_at        timestamptz NOT NULL DEFAULT now(),

  CONSTRAINT portfolio_slug_format
    CHECK (slug ~ '^[a-z0-9][a-z0-9-]*[a-z0-9]$' AND length(slug) BETWEEN 1 AND 80),
  CONSTRAINT portfolio_title_len
    CHECK (length(title) BETWEEN 1 AND 200),
  CONSTRAINT portfolio_description_len
    CHECK (length(description) BETWEEN 1 AND 280),
  CONSTRAINT portfolio_body_len
    CHECK (length(body) BETWEEN 1 AND 50000),
  CONSTRAINT portfolio_outcome_len
    CHECK (outcome IS NULL OR length(outcome) <= 280),
  CONSTRAINT portfolio_project_url_len
    CHECK (project_url IS NULL OR length(project_url) <= 2048),
  CONSTRAINT portfolio_cover_url_len
    CHECK (cover_image_url IS NULL OR length(cover_image_url) <= 2048),
  CONSTRAINT portfolio_tech_stack_is_array
    CHECK (jsonb_typeof(tech_stack) = 'array')
);

CREATE INDEX idx_portfolio_published_sort
  ON portfolio_projects (published, sort_order ASC, created_at DESC);
-- covers: SELECT ... WHERE published = true ORDER BY sort_order, created_at DESC
```

Notes:
- `tech_stack` is a JSONB array of strings, not a separate table — at this scale the joined approach pays no dividends and forces an extra round-trip per detail view.
- Length checks at the DB level give us a safety net if the API layer is bypassed (e.g. ad-hoc SQL). Backend Zod schemas use the same numbers.

### 4.3 `blog_posts`

```sql
CREATE TABLE blog_posts (
  id                uuid        PRIMARY KEY DEFAULT gen_random_uuid(),
  slug              citext      NOT NULL UNIQUE,
  title             text        NOT NULL,
  excerpt           text        NOT NULL,
  body              text        NOT NULL,
  cover_image_url   text        NULL,
  published         boolean     NOT NULL DEFAULT false,
  published_at      timestamptz NULL,                       -- set on first publish, never cleared
  created_at        timestamptz NOT NULL DEFAULT now(),
  updated_at        timestamptz NOT NULL DEFAULT now(),

  CONSTRAINT post_slug_format
    CHECK (slug ~ '^[a-z0-9][a-z0-9-]*[a-z0-9]$' AND length(slug) BETWEEN 1 AND 80),
  CONSTRAINT post_title_len
    CHECK (length(title) BETWEEN 1 AND 200),
  CONSTRAINT post_excerpt_len
    CHECK (length(excerpt) BETWEEN 1 AND 400),
  CONSTRAINT post_body_len
    CHECK (length(body) BETWEEN 1 AND 100000),
  CONSTRAINT post_cover_url_len
    CHECK (cover_image_url IS NULL OR length(cover_image_url) <= 2048),
  CONSTRAINT post_published_implies_publishedat
    CHECK ( (published = false) OR (published_at IS NOT NULL) )
);

CREATE INDEX idx_posts_published_publishedat
  ON blog_posts (published, published_at DESC NULLS LAST);
-- covers: SELECT ... WHERE published = true ORDER BY published_at DESC
```

Notes:
- The `post_published_implies_publishedat` check enforces the invariant the API layer relies on for sorting. The Backend Agent must set `published_at = now()` on the first `false → true` transition, and must leave it alone on subsequent updates.

### 4.4 `tags`

```sql
CREATE TABLE tags (
  id          uuid        PRIMARY KEY DEFAULT gen_random_uuid(),
  slug        citext      NOT NULL UNIQUE,
  name        text        NOT NULL,
  created_at  timestamptz NOT NULL DEFAULT now(),
  updated_at  timestamptz NOT NULL DEFAULT now(),

  CONSTRAINT tag_slug_format
    CHECK (slug ~ '^[a-z0-9][a-z0-9-]*[a-z0-9]$' AND length(slug) BETWEEN 1 AND 40),
  CONSTRAINT tag_name_len
    CHECK (length(name) BETWEEN 1 AND 60)
);
```

### 4.5 `post_tags` (join table)

```sql
CREATE TABLE post_tags (
  post_id   uuid NOT NULL REFERENCES blog_posts(id) ON DELETE CASCADE,
  tag_id    uuid NOT NULL REFERENCES tags(id)       ON DELETE CASCADE,
  PRIMARY KEY (post_id, tag_id)
);

CREATE INDEX idx_post_tags_tag_post
  ON post_tags (tag_id, post_id);
-- supports: "list posts by tag"
```

The composite PK on `(post_id, tag_id)` enforces uniqueness and provides the index for "tags of a given post" (which is the leading-column query). The secondary index supports the reverse — "posts of a given tag" — used by `GET /posts?tag=<slug>`.

### 4.6 `contact_submissions`

```sql
CREATE TYPE contact_status AS ENUM (
  'received',
  'pending_email',
  'emailed',
  'email_failed'
);

CREATE TABLE contact_submissions (
  id              uuid            PRIMARY KEY DEFAULT gen_random_uuid(),
  name            text            NOT NULL,
  email           citext          NOT NULL,
  subject         text            NOT NULL,
  message         text            NOT NULL,
  ip_address      inet            NULL,             -- nullable in case proxy header is missing
  user_agent      text            NULL,
  status          contact_status  NOT NULL DEFAULT 'received',
  email_attempts  smallint        NOT NULL DEFAULT 0,
  last_email_error text           NULL,             -- truncated, max 500 chars
  created_at      timestamptz     NOT NULL DEFAULT now(),

  CONSTRAINT contact_name_len
    CHECK (length(name) BETWEEN 1 AND 100),
  CONSTRAINT contact_subject_len
    CHECK (length(subject) BETWEEN 1 AND 200),
  CONSTRAINT contact_message_len
    CHECK (length(message) BETWEEN 10 AND 5000),
  CONSTRAINT contact_email_attempts_cap
    CHECK (email_attempts >= 0 AND email_attempts <= 10),
  CONSTRAINT contact_email_error_len
    CHECK (last_email_error IS NULL OR length(last_email_error) <= 500)
);

CREATE INDEX idx_contact_status_created
  ON contact_submissions (status, created_at DESC);
-- supports the retry-failed-emails job: WHERE status = 'pending_email' ORDER BY created_at

CREATE INDEX idx_contact_created
  ON contact_submissions (created_at DESC);
-- supports admin list (newest first), unfiltered
```

Notes:
- No `updated_at` because there is no update path from the public side — the retry job writes `status`, `email_attempts`, `last_email_error` but the row is logically append-only from the user perspective.
- `ip_address` uses Postgres `inet` (supports IPv4 + IPv6).
- The status enum is implemented as a real `CREATE TYPE` enum (not a text column with a CHECK) for storage efficiency and to make invalid values impossible.

## 5. Migration sequence

The Database Agent produces these files in `backend/src/db/migrations/`. Order matters — each migration depends on the prior one.

| # | File | Contents |
|---|---|---|
| 000 | `init_extensions.sql` | `pgcrypto`, `citext`, `set_updated_at()` function |
| 001 | `admin_users.sql` | table + trigger |
| 002 | `portfolio_projects.sql` | table + indexes + trigger |
| 003 | `blog_posts.sql` | table + indexes + trigger |
| 004 | `tags.sql` | table + trigger |
| 005 | `post_tags.sql` | join table + index |
| 006 | `contact_submissions.sql` | enum type + table + indexes |

Each migration must have an `up` and a `down`. The `down` for `006` must `DROP TYPE contact_status` AFTER dropping the table (Postgres won't let you drop the type while it's in use).

## 6. Seed data

A single seed runs at API boot, NOT via the migration tool. The Backend Agent owns this logic in `backend/src/db/seed.ts`:

1. If `SELECT count(*) FROM admin_users` returns `0` AND env vars `ADMIN_EMAIL`, `ADMIN_PASSWORD_HASH`, optional `ADMIN_NAME` are set:
   - `INSERT INTO admin_users (email, password_hash, name) VALUES ($1, $2, $3)`.
2. Otherwise no-op.

This idempotent bootstrap is what answers open question Q7. If the founder chooses option (b) instead, the seed file is deleted and replaced with an `npm run create-admin` script.

No seed data for `portfolio_projects`, `blog_posts`, `tags`, or `contact_submissions` — the admin creates real content through the panel. The brief says "no placeholder content in final delivery."

## 7. Indexes summary

| Table | Index | Purpose |
|---|---|---|
| `admin_users` | `UNIQUE (email)` (implicit via UNIQUE) | login lookup |
| `portfolio_projects` | `UNIQUE (slug)` (implicit) | slug lookup |
| `portfolio_projects` | `(published, sort_order ASC, created_at DESC)` | public list |
| `blog_posts` | `UNIQUE (slug)` (implicit) | slug lookup |
| `blog_posts` | `(published, published_at DESC NULLS LAST)` | public list |
| `tags` | `UNIQUE (slug)` (implicit) | tag lookup |
| `post_tags` | `PRIMARY KEY (post_id, tag_id)` | tags of a post |
| `post_tags` | `(tag_id, post_id)` | posts of a tag |
| `contact_submissions` | `(status, created_at DESC)` | retry job |
| `contact_submissions` | `(created_at DESC)` | admin list |

**Intentionally not indexed:** `contact_submissions.email`. We do not query by email at MVP; adding the index would only pay if we built a "submissions by sender" view, which is out of scope.

## 8. Query patterns the Backend Agent will use

Reference list so the Database Agent can `EXPLAIN ANALYZE` against an empty schema and confirm the planned access paths.

```sql
-- list public portfolio
SELECT id, slug, title, description, tech_stack, outcome,
       project_url, cover_image_url, sort_order, created_at, updated_at
  FROM portfolio_projects
 WHERE published = true
 ORDER BY sort_order ASC, created_at DESC
 LIMIT $1 OFFSET $2;

-- portfolio detail
SELECT * FROM portfolio_projects WHERE slug = $1 AND published = true;

-- list public posts with tags
SELECT p.id, p.slug, p.title, p.excerpt, p.cover_image_url, p.published_at,
       COALESCE(
         (SELECT jsonb_agg(jsonb_build_object('slug', t.slug, 'name', t.name))
            FROM post_tags pt JOIN tags t ON t.id = pt.tag_id
           WHERE pt.post_id = p.id), '[]'::jsonb
       ) AS tags
  FROM blog_posts p
 WHERE p.published = true
 ORDER BY p.published_at DESC
 LIMIT $1 OFFSET $2;

-- posts filtered by tag
SELECT p.* , ...
  FROM blog_posts p
  JOIN post_tags pt ON pt.post_id = p.id
  JOIN tags t      ON t.id = pt.tag_id
 WHERE p.published = true AND t.slug = $1
 ORDER BY p.published_at DESC
 LIMIT $2 OFFSET $3;

-- admin login lookup
SELECT id, email, password_hash, name, last_login_at
  FROM admin_users
 WHERE email = $1;

-- contact retry pickup
SELECT id, name, email, subject, message, email_attempts
  FROM contact_submissions
 WHERE status = 'pending_email' AND email_attempts < 3
 ORDER BY created_at ASC
 LIMIT 20;
```

## 10. Future-proofing notes (informational only — do NOT implement at MVP)

The following are anticipated changes the Database Agent should keep in mind when shaping migrations, but should NOT pre-build:

- **Multi-author posts:** add `author_id uuid REFERENCES admin_users(id)` to `blog_posts` (nullable for backfill), then populate from the sole existing admin.
- **Post draft autosave:** add `draft_body text NULL` plus `draft_updated_at` to `blog_posts`.
- **Soft delete:** add `deleted_at timestamptz NULL` to content tables; rewrite indexes with `WHERE deleted_at IS NULL`.
- **Full-text search:** add `tsvector` columns generated from `title || ' ' || body` with GIN indexes.

These are listed so the Database Agent doesn't accidentally close them off (e.g. by making columns NOT NULL where they should be nullable to allow safe future addition).
